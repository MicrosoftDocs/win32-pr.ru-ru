## <a name="remarks"></a><span data-ttu-id="ac5a0-139">Комментарии</span><span class="sxs-lookup"><span data-stu-id="ac5a0-139">Remarks</span></span>

<span data-ttu-id="ac5a0-140">По умолчанию библиотека времени выполнения RPC выполняет вызовы для заданного маркера привязки из каждого потока приложения в определенном порядке отправки.</span><span class="sxs-lookup"><span data-stu-id="ac5a0-140">By default, the RPC run-time library executes the calls on a given binding handle from each thread of an application in strict order of submission.</span></span> <span data-ttu-id="ac5a0-141">Это не гарантирует сериализацию вызовов из разных потоков в одном и том же обработчике привязки.</span><span class="sxs-lookup"><span data-stu-id="ac5a0-141">This does not guarantee that calls from different threads on the same binding handle are serialized.</span></span> <span data-ttu-id="ac5a0-142">Многопоточные приложения должны выполнять сериализацию вызовов RPC.</span><span class="sxs-lookup"><span data-stu-id="ac5a0-142">Multithreaded applications must serialize their RPC calls.</span></span> <span data-ttu-id="ac5a0-143">Если такое поведение является слишком узким, можно включить упорядочение несвязей.</span><span class="sxs-lookup"><span data-stu-id="ac5a0-143">If this behavior is too restrictive, you can enable noncausal ordering.</span></span> <span data-ttu-id="ac5a0-144">В этом случае библиотека времени выполнения RPC выполняет вызовы независимо друг от друга.</span><span class="sxs-lookup"><span data-stu-id="ac5a0-144">When you do, the RPC run-time library executes calls independently.</span></span> <span data-ttu-id="ac5a0-145">При их отправке не будет заказа.</span><span class="sxs-lookup"><span data-stu-id="ac5a0-145">It imposes no ordering on their submission.</span></span>
