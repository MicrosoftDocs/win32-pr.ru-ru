# <a name="why-use-directshow"></a>Зачем использовать DirectShow?

существует две основные причины, по которым приложение может использовать DirectShow, а не пакет SDK Windows Media Format напрямую: для удобства работы с DirectShow архитектурой потоковой передачи и для доступа к оборудованию.

## <a name="convenience"></a>Удобство

с DirectShow архитектурой потоковой передачи для Windows воспроизведения видеофайлов мультимедиа media Audio или Windows media Video требуется всего несколько вызовов метода. Также упрощено создание файлов. вы просто указываете профиль с помощью интерфейса **иконфигасфвритер** в фильтре, и DirectShow автоматически загружает необходимые компоненты для подготовки к просмотру или записи потоков, а также предоставляет механизмы для передачи и синхронизации потока данных мультимедиа. DirectShow особенно полезен при преобразовании содержимого из различных форматов в формат Windows Media. можно создать DirectShow графиков фильтров, которые декодируются разнообразные типы файлов и сжатия, а затем передать декодированные потоки в фильтр [модуля записи WM ASF](wm-asf-writer-filter.md) . По сравнению, пример Ункомпавитовмв в этом пакете SDK работает только с несжатыми AVI файлами. текстовые потоки и произвольные потоки данных также можно создавать и (или) визуализировать с помощью DirectShow, но для этого может потребоваться создать пользовательские фильтры DirectShow для обработки этих потоков.

## <a name="access-to-hardware"></a>Доступ к оборудованию

DirectShow — это единственный способ, которым код приложения получает доступ к устройствам на основе WDM (WDM), таким как 1394 DV-камеры, телевизионные тюнеры и USB-камеры. если приложение должно собирать данные непосредственно с аппаратного устройства на основе WDM и перекодировать его в формат Windows media, а пакет SDK Windows media Encoder не соответствует вашим потребностям, DirectShow является единственной альтернативой. DirectShow также можно использовать для доступа к устаревшим устройствам на основе видео для Windows.
