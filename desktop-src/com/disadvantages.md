# <a name="disadvantages"></a>Недостатки

Внутрипроцессный сервер предоставляет преимущество скорости и размера обработчика объектов с возможностью редактирования локального сервера. Итак, почему вы когда-либо решили реализовать приложение OLE как локальный сервер, а не внутрипроцессный сервер? Существует несколько причин.

-   Безопасность. Только локальный сервер имеет адресное пространство, изолированное от клиента. Внутрипроцессный сервер использует адресное пространство и контекст процесса клиента и, таким образом, может быть менее надежным в случае сбоев или вредоносного программирования.
-   Гранулярности. На локальном сервере может размещаться несколько экземпляров своего объекта на множестве клиентов, совместное использование состояния сервера несколькими объектами в нескольких клиентах, которые могут быть сложными или невозможными, если они реализованы в качестве внутрипроцессного сервера, который является просто библиотекой DLL, загруженной в каждый клиент.
-   См. Если вы решили реализовать внутрипроцессный сервер, вы освобождаете совместимость с OLE 1, который не поддерживает такие серверы. Это не будет рассматриваться для многих разработчиков, но если это так, то это критически важная задача.
-   Невозможность поддержки ссылок. Внутрипроцессный сервер не может выступать в качестве источника ссылки. Так как библиотека DLL не может выполняться сама по себе, она не может создать объект файла, с которым будет связана ссылка.

Несмотря на эти недостатки, внутрипроцессный сервер может быть отличным выбором для его скорости и размера — если он соответствует всем другим требованиям.
