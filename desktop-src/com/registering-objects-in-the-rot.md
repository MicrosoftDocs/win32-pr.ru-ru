# <a name="registering-objects-in-the-rot"></a><span data-ttu-id="97db8-103">Регистрация объектов в таблице ROT</span><span class="sxs-lookup"><span data-stu-id="97db8-103">Registering Objects in the ROT</span></span>

<span data-ttu-id="97db8-104">Как правило, когда клиент запрашивает у сервера создание экземпляра объекта, сервер обычно создает моникер для объекта и регистрирует его в запущенной таблице объектов (ROT) с помощью вызова [**ируннингобжекттабле:: Register**](/windows/desktop/api/ObjIdl/nf-objidl-irunningobjecttable-register).</span><span class="sxs-lookup"><span data-stu-id="97db8-104">Typically, when a client asks a server to create an object instance, the server typically creates a moniker for the object and registers it in the running object table (ROT) through a call to [**IRunningObjectTable::Register**](/windows/desktop/api/ObjIdl/nf-objidl-irunningobjecttable-register).</span></span>

<span data-ttu-id="97db8-105">Когда сервер вызывает [**креатефилемоникер**](/windows/desktop/api/Objbase/nf-objbase-createfilemoniker) для создания специального имени файла для регистрации в таблице ROT, серверы должны передавать локальные имена файлов, которые хранятся на диске, а не в формате UNC.</span><span class="sxs-lookup"><span data-stu-id="97db8-105">When the server calls [**CreateFileMoniker**](/windows/desktop/api/Objbase/nf-objbase-createfilemoniker) to create a file moniker to be registered in the ROT, servers should pass local file names that are drive-based, not in UNC format.</span></span> <span data-ttu-id="97db8-106">Это гарантирует, что данные сравнения моникера, создаваемые вызовом регистрации ROT, будут соответствовать тому, что используется при поиске в таблице ROT для части удаленного клиента.</span><span class="sxs-lookup"><span data-stu-id="97db8-106">This ensures that the moniker comparison data that is generated by the ROT register call will match what is used while doing a ROT lookup on the part of a remote client.</span></span> <span data-ttu-id="97db8-107">Это происходит потому, что когда распределенная служба COM получает запрос на активацию файла, расположенного на локальном сервере, от удаленного клиента, файл преобразуется в путь на локальном диске.</span><span class="sxs-lookup"><span data-stu-id="97db8-107">This is because when the distributed COM service receives an activation request for a file local to the server from a remote client, the file is converted to a local-drive-based path.</span></span>
